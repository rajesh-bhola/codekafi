KsqlDB Queries

Overview





Sometimes, we need to process our events in a real-time stream or by aggregating them together into a materialized table. So queries come into picture.


1. Push Queries


Push a continuous stream of updates to KsqlDB stream or table.

The result of this statement isn't persisted in a kafka topic and is printed out only in the console.

To stop a push query in the CLI, press ctrl + c

Execute a push query via CLI or by sending an HTTP request to the KsqlDB REST API. Keep our push query as simple as possible.

If we need to persist the result of a query to a kafka topic, use a CREATE TABLE AS SELECT or CREATE STREAM AS SELECT statement.

EMIT CHANGES include in KsqlDB 5.4 version onwards.



Example
                
                ksql > SELECT * FROM  pageviews
                       WHERE ROWTIME >= '2017-11-17T04:53:45'
                       EMIT CHANGES;



2. Pull Queries


Pulls the current value from the materialized view and terminates.

The result of this statement isn't persisted in a kafka topic and is printed out only in the console.

Pull queries run with low latency because materialized views are incrementally updated as new events arrived.

Execute a pull query by sending an HTTP request to the KsqlDB REST API.

We can issue a pull query against any table that was created by a "CREATE TABLE AS SELECT" statement.

Do not support - CREATE TABLE, JOIN, PARTITION BY, GROUP BY, WINDOW clauses.



Example
                
                ksql > SELECT * FROM  user_table
                       WHERE ROWKEY = 'user123';
